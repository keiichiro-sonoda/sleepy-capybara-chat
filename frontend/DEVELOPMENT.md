# フロントエンド開発ガイド

## Docker環境での開発

### ホスト設定 (`-H 0.0.0.0`) について

`package.json`の`dev`スクリプトでは以下のように設定しています：

```json
"dev": "next dev -H 0.0.0.0"
```

#### この設定が必要な理由

- **デフォルトの動作**: Next.jsは通常、`localhost`（`127.0.0.1`）のみをリッスンします。これはローカルマシンからのアクセスのみを許可します。

- **Dockerコンテナでの問題**: Dockerコンテナ内でNext.jsを実行する場合、`localhost`だけだとコンテナの外部（ホストマシン）からアクセスできません。これは、Dockerコンテナが独自のネットワーク空間を持っているためです。

- **`-H 0.0.0.0`の役割**: この設定は「すべてのネットワークインターフェース」をリッスンすることを意味し、Dockerコンテナの外部からもアプリケーションにアクセスできるようにします。

- **ポートマッピングとの関係**: `docker-compose.yml`で設定しているポートマッピング（`"3000:3000"`）を有効にするためには、コンテナ内のアプリケーションが`0.0.0.0`をリッスンしている必要があります。

#### 注意点

- この設定はDockerコンテナ内での開発時にのみ必要です。ローカルで直接実行する場合は不要です。
- 本番環境では別の設定が適切な場合があります。

### ホットリロード関連の設定

Dockerコンテナ内でのファイル変更検知（ホットリロード）のために、以下の環境変数を設定しています：

```
CHOKIDAR_USEPOLLING=true
WATCHPACK_POLLING=true
```

- **目的**: Dockerボリュームマウント経由でのファイル変更を検知するため
- **動作**: ファイルシステムの変更イベントの代わりに、定期的なポーリングでファイル変更を検知します
- **デメリット**: CPUリソースの消費が若干増加します

## 環境変数の管理

プロジェクトでは以下の環境変数管理方法を使用しています：

- **開発環境**: `.env.local`ファイルを使用
- **Docker環境**: `.env.local`ファイルをコンテナにボリュームマウント
- **API URL設定**: `NEXT_PUBLIC_API_URL`環境変数でバックエンドAPIのURLを指定

### APIエンドポイントの設定

現在の設定では、`http://localhost:8000/api`をAPIのベースURLとして使用しています。
これにより、フロントエンドからバックエンドのAPIエンドポイントに正しくアクセスできます。

例：
```typescript
// 実際のリクエスト例
await axios.post(`${apiUrl}/v1/auth/verify-email`, ...);
```
